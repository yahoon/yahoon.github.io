介绍#

在我的Home lab里面，我有两台Linux 机器：

  • Ubuntu 26.04 （IP: 192.168.1.4): 作为我的测试服务器，上面运行着我的PoC 环境
  • Fedora 44（IP: 192.168.1.142)：作为我的工作主机

一直以来，他们都工作的非常良好（我讨厌开机，登陆，和重新点开各个应用的等待时间，所以很少关机）。 每天我从我的工作机远程登录到ubuntu 服务器上做各种测试，直到这个周一。

问题描述#

周一的早上，我继续准备从工作机ssh 到ubuntu 服务器上。 我看到之前的会话已经超时断开了，这是预想之中的，不值得大惊小怪。但是当我重新执行历史命令 ssh 192.168.1.4时， 它竟然不动了。我再次确认，ping 也不行了。

我做了如下检查：

  1. 两台机器都可以访问外网
    1. ping baidu.com 和网关 192.168.1.1 =》 OK
    2. 访问外部网页 =》 OK
    3. ip route show 显示路由和出口都正确
  2. 两台机器之间无法互通，ping和ssh都不行
# ping 192.168.1.4
PING 192.168.1.4 (192.168.1.4) 56(84) bytes of data.
From 192.168.1.142 icmp_seq=4 Destination Host Unreachable
ping: sendmsg: No route to host
  1. 两台机器的网络连接正常
    1. ubuntu是有线连接
    2. fedora是无线连接

排错#

防火墙#

第一反应是防火墙问题：

  1. 我把firewall service 完全关闭，但是不起作用

  2. 我想到之前出现过nftables 自定义 rule的问题

# sudo nft list ruleset
table inet mihomo {
	...
}

使用 sudo nft flush table inet mihomo 清理，但是仍然无效

ARP表#

在我的工作机上，我看到arp表有问题

❯ arp -a
? (192.168.1.18) at cc:2d:21:77:42:45 [ether] on wlp9s0
? (192.168.1.4) at <incomplete> on wlp9s0

这说明我根本没有得到对方的ARP应答。网络在二层不通，后续的 ping 就完全没法进行。

检查是否有 arptables 规则在拦截
arptables 是专门用来过滤 ARP 包的，优先级更高。 查看：

# arptables -L
Chain INPUT (policy ACCEPT)

Chain OUTPUT (policy ACCEPT)

检查反向路径过滤 (Reverse Path Filter)

AI说这个是网络参数陷阱。如果严格模式开启，系统会严格检查收到的数据包，看它是否来自最佳的入站网卡。Wi-Fi 漫游或多网卡环境下，这个检查可能出错，导致 ARP 应答包被系统丢弃。

查看并临时关闭它：

# sysctl  -a | grep rp_filter
net.ipv4.conf.all.arp_filter = 0
net.ipv4.conf.all.rp_filter = 0
net.ipv4.conf.default.arp_filter = 0
net.ipv4.conf.default.rp_filter = 2
net.ipv4.conf.eth0.arp_filter = 0
net.ipv4.conf.eth0.rp_filter = 2
net.ipv4.conf.lo.arp_filter = 0
net.ipv4.conf.lo.rp_filter = 2
net.ipv4.conf.wlp9s0.arp_filter = 0
net.ipv4.conf.wlp9s0.rp_filter = 2

# 为无线网卡 wlp9s0 关闭这个过滤
sudo sysctl -w net.ipv4.conf.wlp9s0.rp_filter=0
# 同时关闭全局的
sudo sysctl -w net.ipv4.conf.all.rp_filter=0

刷新 ARP 缓存并测试

设置完后，先清掉 ARP 表里的坏条目，然后立刻重新测试：

# 删掉状态不对的条目
sudo arp -d 192.168.1.4
# 重新 ping 对方
ping 192.168.1.4

还是不行

tcpdump#

# sudo tcpdump -i wlp9s0 arp and host 192.168.1.4
dropped privs to tcpdump
tcpdump: verbose output suppressed, use -v[v]... for full protocol decode
listening on wlp9s0, link-type EN10MB (Ethernet), snapshot length 262144 bytes
21:42:13.359805 ARP, Request who-has 192.168.1.4 tell yahoon-laptop, length 28
21:42:14.394320 ARP, Request who-has 192.168.1.4 tell yahoon-laptop, length 28
21:42:15.418208 ARP, Request who-has 192.168.1.4 tell yahoon-laptop, length 28

21:42:29.626034 ARP, Reply -yahoon-laptop is-at 5e:15:1c:ce:4d:78 (oui Unknown), length 28

网络层面的通信没有任何问题，机器确实收到了 192.168.1.4 发来的 ARP 请求，并且机器也成功发出了 ARP 应答（倒数第二行就是机器在告诉对方“我在这里”）。

但诡异的是，是本机在主动询问对方，而对方其实也在问fedora，双方没有形成正确的“一问一答”配对，导致 ARP 表无法建立。这是一个非常典型的 arp_filter 或 rp_filter 机制在内核层面将 ARP 表项更新逻辑“卡死” 的现象。

静态锁定 ARP 表项（最快验证）

完全绕过动态 ARP 学习，手动把对方的 MAC 地址写死：

# 从之前的 tcpdump 中，看到对方的 MAC 地址是合法的
# 可以再次确认，或者直接执行下面的添加命令
sudo arp -s 192.168.1.4 <对方MAC地址>
ping 192.168.1.4

最后的硬件可能性#

最后我想到无线网卡，有没有是它的问题。

# ethtool -S wlp9s0
NIC statistics:
     rx_packets: 1074
     rx_bytes: 312817
     rx_duplicates: 0
     rx_fragments: 358
     rx_dropped: 43
     tx_packets: 748
     tx_bytes: 396353
     tx_filtered: 0
     tx_retry_failed: 0
     tx_retries: 75
     tx_handlers_drop: 0

这个统计信息非常关键，它把问题从“软件配置”彻底转向了“物理链路质量”。

🎯 真正元凶：无线链路质量太差，回包重传耗尽#

tx_retries: 75 这个数字说明无线网卡在发送数据包时，有 75 次因为没有收到对方的 ACK 确认而进行了重传。对于总共才 748 个发包来说，这个失败率太高了。

这直接解释了为什么会出现 ARP 卡死：当机器试图用单播回应 ARP 请求时，这个数据包因为信号问题发出去后对方收不到，或者对方的 ACK 网卡收不到，导致 ARP 握手在物理层就失败了，最终被上层判定为“目标不可达”。

修复：#

关闭 Wi-Fi 节电模式
节电模式会在网络空闲时降低功率，加剧丢包。请立即执行：

# 查看无线网卡名称（比如 wlp9s0）
iw dev

sudo iw dev wlp9s0 set power_save off
ping 192.168.1.4

最后我通过fedora 桌面做了：

  • 关闭节电模式

反思#

我为了节电， 将电源模式改成了Power-Saver , 这样在机器空闲时，网卡进入了一种非正常的状态

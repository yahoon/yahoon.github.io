在AWS的用户和权限管理都是通过IAM来完成。 IAM中比较复杂的两个概念就是Role 和Policy。本文将我所学习和理解的做一个总结。

AWS IAM管理要点: Role vs Policy#

Role 是什么#

Role 本身只是一个空壳子，它内部包含两部分配置：

  1. 信任策略（Trust Policy）： 决定谁有资格戴上这顶帽子（可以是一台 EC2 服务器，可以是一个 Lambda 函数，也可以是另一个 AWS 账号里的用户）。 这个里面要写principal （谁）
  2. 权限策略（Permission Policy）： 决定戴上这顶帽子后能干什么（这就是把上面介绍的 Policy 绑定过来）。 这个里面写做什么

Policy是什么#

policy就是一个json文件，定义deny 和allow 的动作

IAM 里面提policy默认是指的permission policy

Policy的主元素：#

ElementDescriptionRequired
Effect使用 Allow 或 Deny 表示策略允许或拒绝访问。Yes
Principal指定要允许或拒绝访问的账户、用户、角色或联合用户（仅用于基于资源的策略）。No
Action包含策略允许或拒绝的操作列表。Yes
Resource指定操作所应用的资源列表。No
Condition指定策略授予权限的条件。No

Permission Policy#

最基础和通用的的policy

permission policy 是用来定义权限的

  • 不带principal字段
  • 带action 和resource

例子：

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::我的图片存储桶/*"
    }
  ]
}

Trust policy： role用来定义我代表谁 （或者说谁可以扮演这个role）#

它是用来 定义principal的（也就是谁， 可以是user, service,或者另外一个role）

  • 必须带principal
  • 不带resource
  • Action 总是 sts:AssumeRole 例子
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        // 这里的 Principal 是一个具体的 AWS 服务
        "Service": "lambda.amazonaws.com" 
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

如果你想允许多个同类型或不同类型的主体访问，直接用方括号 [] 组成一个数组。只要满足数组中的任意一个条件，就可以扮演该角色：

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": [
          "arn:aws:iam::111122223333:root",            // 允许账号 111122223333 里的合法用户
          "arn:aws:iam::444455556666:user/ZhangSan"    // 允许账号 444455556666 里的特定用户张三
        ],
        "Service": "lambda.amazonaws.com"              // 同时还允许 Lambda 服务扮演
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

创建 role的过程#

  1. create json file of trust policy（带principal）
  2. create role
  3. create json file of permission policy
  4. attach perm policy to the role

Resource-based Policy: 特殊的policy#

基于资源的policy: 直接在目标res上创建 （不是所有的res都支持）： 直接写具体哪个principal（谁）可以在哪个资源上做什么

基于资源的策略允许您直接从资源本身控制对其的访问权限。并非所有 AWS 服务都支持基于资源的策略，但 Amazon S3 和 Lambda 是支持此功能的两项服务。在本例中，您将配置一个 Lambda 函数，通过使用基于资源的策略来访问 S3 存储桶中的对象。

工作原理：

  1. 将基于资源的策略直接附加到资源上（例如 S3 存储桶策略）。
  2. 指定哪些主体（包括服务使用的 IAM 角色）可以访问它。

注意： Lambda 等 AWS 服务始终需要其服务特定角色才能运行。在同一账户中，基于资源的策略可以授予对 S3 存储桶的访问权限。

详细实现 此 S3 存储桶策略授予 Lambda 执行角色从存储桶读取对象的权限，在资源级别定义访问控制。Lambda 执行角色无需额外 IAM 权限即可访问 Amazon S3。

代码： S3 存储桶资源策略

{
 "Version": "2012-10-17",
 "Statement": [{
	 "Effect": "Allow",
	 "Principal": {
		 "AWS": "arn:aws:iam::123456789012:role/lambda-execution-role"
	 },
	 "Action": "s3:GetObject",
	 "Resource": "arn:aws:s3:::my-bucket/*"
 }]
}

在该策略中：

  • "AWS": "arn:aws:iam::123456789012:role/lambda-execution-role" 指定了 Lambda 执行角色的 ARN。
  • "Resource": "arn:aws:s3:::my-bucket/*" 指定了目标存储桶和对象。

用基于资源的policy 例子： Lamada访问S3#

第一步：在 IAM 中搭建 lambda-execution-role#

你需要创建这顶名叫 lambda-execution-role 的帽子，并明确告诉 AWS：这顶帽子是给 Lambda 服务戴的。

如果你使用 AWS CLI，可以这样搭建：

1. 准备信任策略文件（lambda-trust.json） 由于这个 Role 是要给 Lambda 函数运行用的，它的信任主体（Principal）必须是 Lambda 服务：

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

2. 运行命令创建 Role

aws iam create-role \
    --role-name lambda-execution-role \
    --assume-role-policy-document file://lambda-trust.json

(注：如果你的 AWS 账号 ID 不是 123456789012，创建成功后，它的标准 ARN 会自动变成 arn:aws:iam::<你的真实账号ID>:role/lambda-execution-role)。

3. （可选）为 Role 绑定基础权限 通常 Lambda 需要写日志的权限，可以为它绑定官方的基准策略：

aws iam attach-role-policy \
    --role-name lambda-execution-role \
    --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole

第二步：在 S3 中应用你提供的 Policy#

创建s3的资源访问policy

{
 "Version": "2012-10-17",
 "Statement": [{
	 "Effect": "Allow",
	 "Principal": {
		 "AWS": "arn:aws:iam::123456789012:role/lambda-execution-role"
	 },
	 "Action": "s3:GetObject",
	 "Resource": "arn:aws:s3:::my-bucket/*"
 }]
}

现在扮演者（Role）已经存在了，你需要去目标 S3 存储桶（my-bucket）上开辟绿色通道，允许这个 Role 进来拿东西。

  1. 登录 AWS 控制台，进入 S3 服务，点击进入你的存储桶 my-bucket。

  2. 切换到 Permissions（权限） 选项卡。

  3. 向下滚动找到 Bucket policy（存储桶策略），点击 Edit（编辑）。

  4. 把你题目中提供的那段 JSON 粘贴进去。

  5. 点击 Save changes（保存修改）。

💡 小贴士： 别忘了把 JSON 里的 123456789012 替换为你真实的 AWS 账号 ID（Account ID），否则 S3 会因为找不到这个账号的 Role 而报错。

🚀 终点站：把 Role 绑定到 Lambda 函数#

当你把上面两步都做完后，最后一步就是在创建或配置你的 Lambda 函数时，在 Execution role（执行角色） 设置中，选择你刚刚创建的 lambda-execution-role。

这样一来，闭环就完成了： Lambda 函数启动 ➔ 戴上 lambda-execution-role 面具 ➔ 访问 S3 ➔ S3 的 Bucket Policy 检查发现来者的 ARN 刚好匹配 Principal ➔ 放行并允许下载文件！

传统方式： Lamada访问S3的例子#

通过 Lambda 函数访问 S3 存储桶来解释 Principal 为服务（Service）的场景，是全 AWS 生态中最经典、最直观的例子。

我们可以把这个过程拆解为三步：写剧本（Policy） ➔ 造面具（Role） ➔ 演员登台（Lambda）。

1. 为什么 Lambda 需要一个 Role？#

AWS 的安全原则是默认拒绝一切。一个刚创建出来的 Lambda 函数是一张白纸，它默认没有任何权限，根本摸不到你的 S3 存储桶。

为了让 Lambda 能去读写 S3，我们必须为它量身定做一顶帽子（IAM Role）。

2. 核心配置：拆解这个 Role 的两个policy#

当我们配置这个给 Lambda 使用的 Role 时，我们需要在 IAM 中配置两个关键部分: trust policy & permission policy：

信任策略（Trust Policy）#

规定“谁能戴这顶帽子”

在这里，Principal 就是 AWS Lambda 服务本身。这份策略告诉 IAM：“我只允许 Lambda 这个服务来扮演我，别的人或服务（比如 EC2 或人类用户）休想碰我。”

其 JSON 结构如下：

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        // 这里的 Principal 是一个具体的 AWS 服务
        "Service": "lambda.amazonaws.com" 
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

权限策略（Permission Policy）#

—— 规定“戴上后能干什么”

这就是一份普通的 IAM Policy。我们把它绑定到这个 Role 上，规定戴上帽子后可以对 S3 做什么。

其 JSON 结构如下：

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::我的图片存储桶/*"
    }
  ]
}

步骤#

第一步：创建只有 Trust Policy 的空壳角色#

aws iam create-role \
    --role-name lambda-execution-role \
    --assume-role-policy-document file://lambda-trust.json

💡 执行完这一步，这个 Role 在 AWS 确实只有 Trust Policy。此时它还是一顶没有任何法力的“空帽子”。

第二步：紧接着把 Permission Policy 贴上去（这一步就是塞权限）#

我这里先塞入一个默认权限

aws iam attach-role-policy \
    --role-name lambda-execution-role \
    --policy-arn arn:aws:policy/service-role/AWSLambdaBasicExecutionRole

💡 这一步就是往帽子里塞入权限。我当时示例里绑定的是 AWS 官方的 AWSLambdaBasicExecutionRole（里面包含写日志的权限）。

第三步： 应用S3访问的 permission policy#

这个 Role 内部明明白白地带上 S3 的下载权限（Permission Policy），我们在执行完第一步创建空壳后，应该紧接着执行下面这一步：

1. 准备你的权限文件（s3-permission.json）：

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}

2. 把这个自定义权限直接“拍”给刚刚建好的 Role：

aws iam put-role-policy \
    --role-name lambda-execution-role \
    --policy-name LambdaS3DownloadPolicy \
    --policy-document file://s3-permission.json

完整的运行工作流（运行时发生了什么？）#

当你的 Lambda 函数被触发执行（比如有人上传了一张图片到 S3，触发了 Lambda 去处理）时，底层的运作逻辑是这样的：

[Lambda 服务] ➔ (戴上/扮演) ➔ [IAM Role] ➔ (获得临时凭证) ➔ [读取 S3 存储桶]

  1. 服务代入角色（AssumeRole）： AWS Lambda 服务看到这个函数绑定了你刚才建的 Role，它就会向 AWS STS（安全令牌服务）发起请求：“我是 lambda.amazonaws.com（符合 Principal 验证），我现在要扮演这个 Role”。

  2. 分发临时令牌： AWS 验证通过后，会生成一组只有几分钟到几小时有效期的临时 Access Key 和 Session Token，并把这组凭证偷偷塞进这个正在运行的 Lambda 代码环境变量里。

  3. 安全访问资源： 你的 Lambda 代码（无论是用 Python 还是 Node.js 写得 boto3.client('s3')）在执行时，会自动读取这组临时凭证，成功把文件写入或读出 S3。

  4. 人走茶凉： 运行完毕后，Lambda 容器销毁，临时凭证瞬间失效。下一次再触发时，重新重复这一套流程。

这就是为什么你在写 Lambda 访问 S3 的代码时，不需要在代码里写任何 AWS_ACCESS_KEY。因为 Principal: lambda.amazonaws.com 这行配置，已经让 Lambda 服务本身具备了自动去 IAM 换取临时钥匙的合法特权。

业界标准做法：永远双向配置#

在实际的生产环境中，没有人会去赌“单边授权”的特例。为了让架构清晰、可维护，标准的做法是两边都写：

  1. 在 Role 上： 既有 Trust Policy（允许 Lambda 戴帽子），又有 Permission Policy（赋予它 s3:GetObject 权限）。

  2. 在 S3 桶上： 保持默认，或者如果需要极高安全性，再用 Bucket Policy 锁死只允许这个 Role 进。

这样哪怕以后这个 S3 存储桶的规则改了，或者 Lambda 需要访问另一个没有配置桶策略的普通 S3 桶，你的 Lambda 程序都不会莫名其妙地崩溃。

所以，这顶帽子（Role）在被造出来的时候，我们还是习惯拍拍它的脑袋，往里面塞一张写着 s3:GetObject 的纸条（Permission Policy），让它出远门时心里更有底气。

1. 如果你只依赖“资源策略（Bucket Policy）”#

假设你采取了最极端的做法：Role 里面没有任何 S3 权限（一片空白），完全靠 my-bucket 的 Bucket Policy 单方面给这个 Role 授权。

  • 隐患： 某一天，你的业务升级了，Lambda 除了需要读 my-bucket，还需要顺便去读一下 public-data-bucket（这是一个别人公开的、或者公司内部公共的、你没有权限去修改其桶策略的普通桶）。

  • 崩溃原因： 因为你无法去修改那个公共桶的 Bucket Policy，而你的 Role 自身又是一片空白，你的 Lambda 程序就会在访问新桶时彻底崩溃。

2. 为什么说“在 Role 里配权限”更健壮？#

如果我们按照标准做法，在 Role 的 Permission Policy 里去管理权限：

  • 场景： 同样是要访问新桶 public-data-bucket。

  • 如何解决： 你完全不需要动 S3 端的任何规则。你只需要来到 IAM，修改这个 Role 自己的 Permission Policy，在 Resource 列表里加上新桶的 ARN：

    "Resource": [
  "arn:aws:s3:::my-bucket/*",
  "arn:aws:s3:::public-data-bucket/*"
]

结果： 权限的修改完全在 IAM（身份端）闭环完成。你不需要去求着公共桶的管理员帮你改桶策略，你的 Lambda 就能顺利访问新桶。

💡 总结 AWS 权限控制的“黄金法则”#

正如你所说，如果 Permission Policy 里写死了具体的桶，换桶肯定会拒绝。但两者的核心区别在于**“改错的成本和控制权”**：

  1. 改 IAM Role 的 Permission Policy： 属于“修改 Lambda 这个人手里的通行证”。通行证在你自己手里，你想去哪，自己加个地址（新桶 ARN）就能去，非常灵活。
  2. 改 S3 的 Bucket Policy： 属于“修改特定大楼的门卫规则”。如果哪天大楼换了、或者你要去别的大楼，你没办法命令别的大楼也为你修改门卫规则。

所以，在编写 Role 的 Permission Policy 时：

  • 如果为了极致的安全，应该像你一样，精确写死具体的桶 arn:aws:s3:::my-bucket/*。
  • 如果为了通用性的自动化工具（比如一个通用的日志收集 Lambda），大家有时会把 Role 的权限放宽到 "Resource": "arn:aws:s3:::*"（允许访问所有桶），然后利用具体的子网 VPC 或者 S3 桶策略在资源端做最后的审计。

附录： AWS Policy的分类#

AWS policy 分为多种：

  • trust policy
  • permission policy
  • resource-based policy

只有trust policy 和 resource-based policy 需要带principal.

在 AWS 的安全世界里，各种 Policy 确实因为“挂载的地方不同”而分成了不同的种类。关于 Principal（主体/你是谁）的规律，正如你所说：只有当这份 Policy 需要回答“谁来访问”时，它才需要带 Principal。

我们可以用一张清晰的表格来横向对比它们，彻底帮你闭环这个知识点：

Policy 种类挂载/贴在什么地方？它的核心任务是回答什么？❌ 是否需要 Principal？
Trust Policy
（信任策略）		贴在 IAM Role（角色） 上“谁有资格扮演我这顶帽子？”必须带。（比如：允许某 Lambda 服务或某 User 扮演）
Resource-based Policy
（资源策略）		贴在 AWS 资源上
（如 S3 存储桶、SQS 队列）		“谁有资格进入我的地盘拿东西？”必须带。（因为资源在自己的地盘上，必须指名道姓谁能进来）
Permission Policy
（权限/身份策略）		贴在 IAM User（用户） 或 Role（角色） 内部“我作为这个身份，出门能干什么？”绝对不要带。（因为已经贴在具体人/角色身上了，主体就是它自己）

🧠 为什么 Permission Policy 绝对不要带 Principal？#

为了加深记忆，我们来看如果不遵守这个规律会发生什么。

假设你在一个名叫 ZhangSan 的 IAM 用户身上，贴了一张 Permission Policy。如果语法允许你写 Principal: LiSi，那 AWS 的系统就会彻底陷入混乱：这到底是在规定张三的权限，还是在规定李四的权限？

所以，AWS 的设计非常严格：

  • 只要是“主动型”的身份（User, Role），它们身上带的 Permission Policy 默认主语就是它们自己，所以禁止写 Principal。
  • 只有“被动型”的角色壳子（Trust Policy）和“被动型”的资源（Resource Policy），它们需要开门迎客，才必须在 Policy 里加上 Principal 来规定“谁能进来”。

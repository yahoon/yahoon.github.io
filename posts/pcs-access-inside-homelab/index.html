Introduction#

In my home lab, I have two Linux machines:

  • Ubuntu 26.04 (IP: 192.168.1.4): acts as my test server, running my PoC environment
  • Fedora 44 (IP: 192.168.1.142): acts as my workstation

They have been working perfectly well (I hate the waiting time for booting, logging in, and reopening applications, so I rarely shut them down). Every day I SSH from my workstation to the Ubuntu server for various tests, until this Monday.

Problem Description#

On Monday morning, I continued to prepare to SSH from my workstation to the Ubuntu server. I saw the previous session had timed out, which was expected and no big deal. But when I re-executed the history command ssh 192.168.1.4, it just hung. I double-checked, and ping didn’t work either.

I performed the following checks:

  1. Both machines could access the external network
    1. ping baidu.com and gateway 192.168.1.1 => OK
    2. Accessing external web pages => OK
    3. ip route show showed correct routing and gateway
  2. The two machines could not communicate with each other, neither ping nor ssh worked
# ping 192.168.1.4
PING 192.168.1.4 (192.168.1.4) 56(84) bytes of data.
From 192.168.1.142 icmp_seq=4 Destination Host Unreachable
ping: sendmsg: No route to host
  1. Both machines’ network connections were normal
    1. Ubuntu used wired connection
    2. Fedora used wireless connection

Troubleshooting#

Firewall#

First reaction was a firewall issue:

  1. I completely disabled the firewall service, but it didn’t help

  2. I remembered a previous issue with custom nftables rules

# sudo nft list ruleset
table inet mihomo {
	...
}

Cleared it with sudo nft flush table inet mihomo, but still no effect

ARP Table#

On my workstation, I saw an issue with the ARP table

❯ arp -a
? (192.168.1.18) at cc:2d:21:77:42:45 [ether] on wlp9s0
? (192.168.1.4) at <incomplete> on wlp9s0

This meant I never received the ARP reply from the other machine. The network was broken at Layer 2, so subsequent ping couldn’t work at all.

Check if arptables rules are blocking

arptables is specifically used to filter ARP packets and has higher priority. Check:

# arptables -L
Chain INPUT (policy ACCEPT)

Chain OUTPUT (policy ACCEPT)

Check Reverse Path Filter

AI said this is a network parameter pitfall. If strict mode is enabled, the system strictly checks incoming packets to see if they come from the best inbound interface. In Wi-Fi roaming or multi-NIC environments, this check can go wrong, causing ARP reply packets to be dropped by the system.

Check and temporarily disable it:

# sysctl  -a | grep rp_filter
net.ipv4.conf.all.arp_filter = 0
net.ipv4.conf.all.rp_filter = 0
net.ipv4.conf.default.arp_filter = 0
net.ipv4.conf.default.rp_filter = 2
net.ipv4.conf.eth0.arp_filter = 0
net.ipv4.conf.eth0.rp_filter = 2
net.ipv4.conf.lo.arp_filter = 0
net.ipv4.conf.lo.rp_filter = 2
net.ipv4.conf.wlp9s0.arp_filter = 0
net.ipv4.conf.wlp9s0.rp_filter = 2

# Disable this filter for wireless interface wlp9s0
sudo sysctl -w net.ipv4.conf.wlp9s0.rp_filter=0
# Also disable globally
sudo sysctl -w net.ipv4.conf.all.rp_filter=0

Flush ARP cache and test

After setting, clear the bad entries from the ARP table and immediately retest:

# Delete the entry with bad state
sudo arp -d 192.168.1.4
# Re-ping the target
ping 192.168.1.4

Still didn’t work

tcpdump#

# sudo tcpdump -i wlp9s0 arp and host 192.168.1.4
dropped privs to tcpdump
tcpdump: verbose output suppressed, use -v[v]... for full protocol decode
listening on wlp9s0, link-type EN10MB (Ethernet), snapshot length 262144 bytes
21:42:13.359805 ARP, Request who-has 192.168.1.4 tell yahoon-laptop, length 28
21:42:14.394320 ARP, Request who-has 192.168.1.4 tell yahoon-laptop, length 28
21:42:15.418208 ARP, Request who-has 192.168.1.4 tell yahoon-laptop, length 28
21:42:29.625986 ARP, Request who-has -yahoon-laptop tell 192.168.1.4, length 46
21:42:29.626034 ARP, Reply -yahoon-laptop is-at 5e:15:1c:ce:4d:78 (oui Unknown), length 28

There was no problem at the network layer communication. The machine did receive the ARP request from 192.168.1.4, and the machine successfully sent an ARP reply (the second-to-last line shows the machine telling the other side “I am here”).

But the strange thing was that the local machine was actively querying the other side, while the other side was also querying Fedora, and the two sides failed to form a proper Q&A pairing, causing the ARP table to never be established. This was a very typical symptom of the arp_filter or rp_filter mechanism “deadlocking” the ARP table update logic at the kernel level.

Static ARP entry (fastest verification)

Completely bypass dynamic ARP learning and manually hardcode the other machine’s MAC address:

# From the previous tcpdump, the other machine's MAC address is valid
# Double-check or directly execute the following command
sudo arp -s 192.168.1.4 <target MAC address>
ping 192.168.1.4

The final hardware possibility#

Finally, I wondered if the wireless network card was the problem.

# ethtool -S wlp9s0
NIC statistics:
     rx_packets: 1074
     rx_bytes: 312817
     rx_duplicates: 0
     rx_fragments: 358
     rx_dropped: 43
     tx_packets: 748
     tx_bytes: 396353
     tx_filtered: 0
     tx_retry_failed: 0
     tx_retries: 75
     tx_handlers_drop: 0

This statistic was crucial. It shifted the issue entirely from “software configuration” to “physical link quality.”

🎯 The Real Culprit: Poor Wireless Link Quality, Retransmission Exhaustion#

tx_retries: 75 indicated that when the wireless NIC sent data packets, there were 75 retransmissions due to not receiving ACK confirmation from the other side. For only 748 total transmitted packets, this failure rate was too high.

This directly explained why ARP got stuck: when the machine tried to respond to an ARP request via unicast, the packet was sent but couldn’t reach the other side due to signal issues, or the other side’s ACK couldn’t be received by the NIC. This caused the ARP handshake to fail at the physical layer, ultimately being judged by the upper layer as “destination unreachable.”

Fix:#

Disable Wi-Fi Power Saving Mode Power saving mode reduces power during idle periods, exacerbating packet loss. Execute immediately:

# Check wireless interface name (e.g., wlp9s0)
iw dev

sudo iw dev wlp9s0 set power_save off
ping 192.168.1.4

Finally, I did this through the Fedora desktop:

  • Disabled power saving mode

Reflection#

To save power, I had changed the power mode to Power-Saver, which caused the network card to enter an abnormal state when the machine was idle.

User and permission management in AWS is done through IAM. The two more complex concepts in IAM are Role and Policy. This article summarizes what I have learned and understood.

AWS IAM Key Points: Role vs Policy#

What is a Role#

A Role itself is just an empty shell; it contains two parts of configuration inside:

  1. Trust Policy: Determines who is eligible to wear this hat (it could be an EC2 server, a Lambda function, or a user in another AWS account). This is where you specify the principal (who).
  2. Permission Policy: Determines what actions can be performed while wearing this hat (this binds the Policy introduced above). This is where you specify what actions.

What is a Policy#

A policy is a JSON file that defines deny and allow actions.

When referring to policy in IAM, it defaults to permission policy.

Main Elements of a Policy:#

ElementDescriptionRequired
EffectUse Allow or Deny to indicate whether the policy allows or denies access.Yes
PrincipalIndicate the account, user, role, or federated user to which you would like to allow or deny access (only on resource policies).No
ActionInclude a list of actions that the policy allows or denies.Yes
ResourceSpecify a list of resources to which the actions apply.No
ConditionSpecify the circumstances under which the policy grants permission.No

Permission Policy#

The most basic and universal policy.

Permission policy is used to define permissions.

  • Does not include the principal field
  • Includes action and resource

Example:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::我的图片存储桶/*"
    }
  ]
}

Trust policy: role is used to define who I represent (or who can assume this role)#

It is used to define the principal (i.e., who, which can be a user, service, or another role).

  • Must include principal
  • Does not include resource
  • Action is always sts:AssumeRole

Example:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        // Here Principal is a specific AWS service
        "Service": "lambda.amazonaws.com" 
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

If you want to allow multiple principals of the same or different types to access, simply use square brackets [] to form an array. As long as any one condition in the array is met, the role can be assumed:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": [
          "arn:aws:iam::111122223333:root",            // Allows legitimate users in account 111122223333
          "arn:aws:iam::444455556666:user/ZhangSan"    // Allows specific user ZhangSan from account 444455556666
        ],
        "Service": "lambda.amazonaws.com"              // Also allows Lambda service to assume
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

Process of Creating a Role#

  1. Create a JSON file for the trust policy (includes principal)
  2. Create the role
  3. Create a JSON file for the permission policy
  4. Attach the permission policy to the role

Resource-based Policy: A Special Policy#

Resource-based policy: Created directly on the target resource (not supported by all resources): directly specify which principal (who) can do what on which resource.

Resource policies give you the ability to control access to AWS resources directly from the resource itself. Not all AWS services support resource policies, but Amazon S3 and Lambda are two services that do. In this example, you configure a Lambda function to access objects in an S3 bucket by using resource policies.

How it works:

  1. Attach a resource policy directly to the resource (such as an S3 bucket resource policy).
  2. Specify which principals (including IAM roles used by services) can access it.

Note: AWS services such as Lambda always need their service-specific roles to operate. In the same account, resource-based policies can grant access to the S3 bucket.

Detailed implementation This S3 bucket policy grants the Lambda execution role permission to read objects from the bucket, defining access at the resource level. No additional IAM permissions are needed on the Lambda execution role for Amazon S3 access.

Code: S3 Bucket Resource Policy

{
 "Version": "2012-10-17",
 "Statement": [{
	 "Effect": "Allow",
	 "Principal": {
		 "AWS": "arn:aws:iam::123456789012:role/lambda-execution-role"
	 },
	 "Action": "s3:GetObject",
	 "Resource": "arn:aws:s3:::my-bucket/*"
 }]
}

In the policy:

  • “AWS”: arn:aws:iam::123456789012:role/lambda-execution-role"" specifies the lambda execution role ARN.
  • “Resource”: “arn:aws:s3:::my-bucket/*” specifies the target bucket and objects.

Example of Using Resource-based Policy: Lambda Accessing S3#

Step 1: Create lambda-execution-role in IAM#

You need to create this hat named lambda-execution-role, and clearly tell AWS: this hat is for the Lambda service to wear.

If you use AWS CLI, you can set it up like this:

1. Prepare the trust policy file (lambda-trust.json) Since this Role is for Lambda function execution, its trust principal must be the Lambda service:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

2. Run command to create Role

aws iam create-role \
    --role-name lambda-execution-role \
    --assume-role-policy-document file://lambda-trust.json

(Note: If your AWS account ID is not 123456789012, after successful creation, its standard ARN will automatically become arn:aws:iam::<your-real-account-ID>:role/lambda-execution-role).

3. (Optional) Attach basic permissions to the Role Usually Lambda needs permission to write logs; you can attach the official baseline policy to it:

aws iam attach-role-policy \
    --role-name lambda-execution-role \
    --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole

Step 2: Apply the Policy to S3#

Create S3 resource access policy:

{
 "Version": "2012-10-17",
 "Statement": [{
	 "Effect": "Allow",
	 "Principal": {
		 "AWS": "arn:aws:iam::123456789012:role/lambda-execution-role"
	 },
	 "Action": "s3:GetObject",
	 "Resource": "arn:aws:s3:::my-bucket/*"
 }]
}

Now that the role (Role) exists, you need to open a green channel on the target S3 bucket (my-bucket) to allow this Role to come in and retrieve objects.

  1. Log into the AWS console, go to the S3 service, click on your bucket my-bucket.
  2. Switch to the Permissions tab.
  3. Scroll down to find Bucket policy, click Edit.
  4. Paste the JSON provided in your policy into it.
  5. Click Save changes.

💡 Tip: Don’t forget to replace 123456789012 in the JSON with your real AWS account ID (Account ID), otherwise S3 will throw an error because it can’t find the Role for this account.

🚀 Final Step: Bind the Role to the Lambda Function#

After completing the above two steps, the final step is to select the lambda-execution-role you just created in the Execution role settings when creating or configuring your Lambda function.

This completes the loop: Lambda function starts ➔ wears the lambda-execution-role mask ➔ accesses S3 ➔ S3’s Bucket Policy checks and finds the visitor’s ARN matches Principal ➔ allows access and downloads the file!

Traditional Approach: Lambda Accessing S3 Example#

Using Lambda function accessing S3 bucket to explain the scenario where Principal is a service (Service) is the most classic and intuitive example in the entire AWS ecosystem.

We can break down this process into three steps: Write the script (Policy) ➔ Make the mask (Role) ➔ Actor takes the stage (Lambda).

1. Why does Lambda need a Role?#

AWS’s security principle is deny everything by default. A newly created Lambda function is a blank slate; it has no permissions by default and cannot access your S3 bucket at all.

To allow Lambda to read and write to S3, we must tailor a hat (IAM Role) for it.

2. Core Configuration: Breaking Down the Two Policies of this Role#

When configuring this Role for Lambda use, we need to configure two key parts in IAM: trust policy & permission policy:

Trust Policy#

Specifies “who can wear this hat”

Here, Principal is the AWS Lambda service itself. This policy tells IAM: “I only allow the Lambda service to assume me; no other people or services (like EC2 or human users) can touch me.”

Its JSON structure is as follows:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        // Here Principal is a specific AWS service
        "Service": "lambda.amazonaws.com" 
      },
      "Action": "sts:AssumeRole"
    }
  ]
}

Permission Policy#

—— Specifies “what can be done after wearing it”

This is a standard IAM Policy. We attach it to this Role to specify what actions can be performed on S3 while wearing the hat.

Its JSON structure is as follows:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::我的图片存储桶/*"
    }
  ]
}

Steps#

Step 1: Create an Empty Shell Role with Only Trust Policy#

aws iam create-role \
    --role-name lambda-execution-role \
    --assume-role-policy-document file://lambda-trust.json

💡 After executing this step, this Role in AWS indeed only has Trust Policy. At this point, it is still an “empty hat” with no power.

Step 2: Attach the Permission Policy (This step adds permissions)#

Here I’ll first attach a default permission:

aws iam attach-role-policy \
    --role-name lambda-execution-role \
    --policy-arn arn:aws:policy/service-role/AWSLambdaBasicExecutionRole

💡 This step is what it means to put permissions into the hat. In my example, I attached the official AWS AWSLambdaBasicExecutionRole (which includes permission to write logs).

Step 3: Apply the S3 Access Permission Policy#

This Role explicitly carries S3 download permissions (Permission Policy). After completing the first step of creating the empty shell, we should immediately execute the following step:

1. Prepare your permission file (s3-permission.json):

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}

2. Attach this custom permission directly to the newly created Role:

aws iam put-role-policy \
    --role-name lambda-execution-role \
    --policy-name LambdaS3DownloadPolicy \
    --policy-document file://s3-permission.json

Complete Runtime Workflow (What happens at runtime?)#

When your Lambda function is triggered to execute (for example, someone uploads an image to S3, triggering Lambda to process it), the underlying operational logic is as follows:

[Lambda Service] ➔ (assumes/wears) ➔ [IAM Role] ➔ (gets temporary credentials) ➔ [Reads S3 bucket]
  1. Service AssumeRole: The AWS Lambda service sees that this function is bound to the Role you just created, and it will send a request to AWS STS (Security Token Service): “I am lambda.amazonaws.com (passes Principal validation), and I now want to assume this role.”
  2. Distribute Temporary Credentials: After AWS validates the request, it generates a set of temporary Access Key and Session Token valid for only a few minutes to hours, and quietly injects these credentials into the running Lambda code’s environment variables.
  3. Securely Access Resources: Your Lambda code (whether written in Python or Node.js as boto3.client('s3')) automatically reads these temporary credentials during execution and successfully writes or reads files from S3.
  4. Cleanup: After execution completes, the Lambda container is destroyed, and the temporary credentials become invalid instantly. The next time it’s triggered, the entire process repeats.

This is why, when writing Lambda code to access S3, you don’t need to write any AWS_ACCESS_KEY in your code. Because the Principal: lambda.amazonaws.com configuration already gives the Lambda service itself the legal privilege to automatically exchange for temporary keys from IAM.

Industry Standard Practice: Always Configure Both Sides#

In actual production environments, no one bets on the “single-sided authorization” exception. To keep the architecture clean and maintainable, the standard practice is to configure both sides:

  1. On the Role: Both Trust Policy (allowing Lambda to wear the hat) and Permission Policy (granting it s3:GetObject permission).
  2. On the S3 bucket: Keep the default, or if extremely high security is needed, use Bucket Policy to lock it down to only allow this Role.

This way, even if the S3 bucket’s rules change later, or Lambda needs to access another ordinary S3 bucket without a bucket policy configured, your Lambda program won’t crash inexplicably.

So, when this hat (Role) is created, we still habitually pat its head and stuff a note with s3:GetObject written on it (Permission Policy) inside, so it feels more confident when going out.

1. If You Only Rely on “Resource Policy (Bucket Policy)”#

Suppose you take the most extreme approach: The Role has no S3 permissions (completely blank), relying entirely on my-bucket’s Bucket Policy to unilaterally authorize this Role.

  • Risk: One day, your business upgrades, and Lambda needs to read public-data-bucket in addition to my-bucket (this is a publicly accessible bucket, or an internal company bucket where you don’t have permission to modify its bucket policy).
  • Reason for Failure: Because you can’t modify that public bucket’s Bucket Policy, and your Role itself is blank, your Lambda program will completely crash when accessing the new bucket.

2. Why is “Configuring Permissions in the Role” More Robust?#

If we follow the standard practice of managing permissions in the Role’s Permission Policy:

  • Scenario: Also needing to access the new bucket public-data-bucket.

  • Solution: You don’t need to touch any S3-side rules. You simply go to IAM, modify this Role’s own Permission Policy, and add the new bucket’s ARN to the Resource list:

    "Resource": [
  "arn:aws:s3:::my-bucket/*",
  "arn:aws:s3:::public-data-bucket/*"
]

Result: Permission changes are completely self-contained within IAM (identity side). You don’t need to beg the public bucket’s administrator to modify the bucket policy; your Lambda can smoothly access the new bucket.

💡 Summary: The “Golden Rule” of AWS Permission Control#

As you said, if the Permission Policy hardcodes a specific bucket, changing buckets will certainly be denied. But the core difference between the two lies in “cost of modification and control”:

  1. Modify IAM Role’s Permission Policy: This is like “modifying the pass in Lambda’s hand.” The pass is in your own hands; wherever you want to go, you just add the address (new bucket ARN) yourself, very flexible.
  2. Modify S3’s Bucket Policy: This is like “modifying the building’s security guard rules.” If the building changes one day, or you need to go to another building, you can’t command another building to modify its security guard rules for you.

Therefore, when writing a Role’s Permission Policy:

  • If aiming for extreme security, you should precisely hardcode the specific bucket arn:aws:s3:::my-bucket/* as you did.
  • If aiming for general-purpose automation tools (like a generic log collection Lambda), people sometimes relax the Role’s permissions to "Resource": "arn:aws:s3:::*" (allow access to all buckets), then use specific VPC subnets or S3 bucket policies for final auditing at the resource side.

Appendix: Classification of AWS Policies#

AWS policies are divided into several types:

  • trust policy
  • permission policy
  • resource-based policy

Only trust policy and resource-based policy need to include principal.

In the AWS security world, various policies are indeed categorized by “where they are attached.” Regarding the rule about Principal (who you are), as you said: Only when this Policy needs to answer “who is accessing” does it need to include Principal.

We can use a clear table to compare them horizontally, thoroughly helping you close the loop on this knowledge point:

Policy TypeWhere is it attached?Its core task is to answer what?Does it need Principal?
Trust PolicyAttached to IAM Role“Who is qualified to assume this hat?”Must include. (e.g., allow a Lambda service or User to assume)
Resource-based PolicyAttached to AWS Resource (e.g., S3 bucket, SQS queue)“Who is qualified to enter my territory to get things?”Must include. (Because the resource is on its own territory, it must specify who can enter)
Permission Policy (Identity Policy)Attached internally to IAM User or Role“What can I do when I go out as this identity?”Must NOT include. (Because it’s already attached to a specific person/role, the subject is itself)

🧠 Why Should Permission Policy NEVER Include Principal?#

To reinforce this, let’s see what happens if we don’t follow this rule.

Suppose you attach a Permission Policy to an IAM user named ZhangSan. If the syntax allowed you to write Principal: LiSi, the AWS system would become completely confused: Is this defining ZhangSan’s permissions or LiSi’s permissions?

Therefore, AWS’s design is very strict:

  • For “active” identities (User, Role), the Permission Policy attached to them defaults to them as the subject, so writing Principal is prohibited.
  • Only “passive” role shells (Trust Policy) and “passive” resources (Resource Policy), which need to open the door to guests, must include Principal in the Policy to specify “who can come in.”
